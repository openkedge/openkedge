# RFC-0002: Event Evidence Chain (EEC)

**Status:** Draft  
**Author:** Jun He  
**Project:** OpenKedge  
**Date:** April 2026  
**Scope:** Verifiable Event Logging and State Reconstruction

---

## 1. Abstract

This document defines the Event Evidence Chain (EEC), an append-only, deterministic event model for recording all intent-driven mutations governed by the Intent Governance Protocol (IGP, [RFC-0001]).

The EEC establishes a verifiable, replayable history of autonomous decisions by capturing the full lifecycle of each mutation: intent submission, context evaluation, policy enforcement, execution, and resulting state transitions. By modeling state as a strict reduction of immutable events, the EEC transforms system mutation from an ephemeral, untraceable action into a persistent, cryptographically auditable sequence of evidence-backed decisions.

## 3. Problem Statement

Existing infrastructure systems and orchestration layers execute mutations without preserving the complete decision context. In agentic systems, this leads to critical failures:

* **Loss of Reasoning Trace:** The "why" behind an autonomous action is discarded post-execution.
* **Irreproducibility:** Inability to deterministically replay system behavior for debugging or simulation.
* **Audit Deficiencies:** Lack of verifiable evidence for compliance and safety audits.
* **Multi-Agent Drift:** Inconsistent interpretation of the system's history across distributed agents.

These limitations prevent reliable coordination, safety enforcement, and post-hoc verification, making autonomous infrastructure inherently fragile.

---

## 4. Terminology

* **Event:** A single immutable record describing a distinct stage in the IGP mutation lifecycle.
* **Evidence Chain:** An ordered, hash-linked sequence of Events representing the absolute truth of system evolution.
* **Replay:** The deterministic process of reconstructing system state and decisions from the Evidence Chain.
* **Determinism:** The mathematical property that identical context and intents MUST produce identical policy and state outputs.
* **Context Snapshot:** An immutable capture of the system state at the exact moment of policy evaluation.
* **Decision Record:** The explicit ALLOW/DENY output generated by a Policy Engine.

---

## 5. Model Overview

The Event Evidence Chain models system evolution as a directed, acyclic sequence of events:

$E_0 \rightarrow E_1 \rightarrow E_2 \rightarrow \dots \rightarrow E_n$

Each event corresponds directly to a state transition defined in IGP (RFC-0001). The absolute system state at any given discrete step, $S_n$, MUST be calculable via a deterministic reduction (fold) function over the event history:

$$S_n = \text{fold}(E_0, E_1, \dots, E_n)$$

Where the `fold` function is a pure, deterministic state-accumulator.

---

## 6. Specification: Event Types

To satisfy the IGP lifecycle, implementations MUST support the following standardized event types:

* **`IntentSubmitted`:** The initial declaration of a desired mutation, signed by the requesting actor.
* **`ContextEvaluated`:** The capture of the Context Snapshot used as input for the policy engine.
* **`PolicyEvaluated`:** The ALLOW/DENY output from the policy engine, mapping the Intent and Context to a Decision Record.
* **`ExecutionStarted`:** The minting of the Bounded Execution Identity and initiation of the mutation.
* **`ExecutionCompleted`:** The successful return payload from the infrastructure API.
* **`ExecutionFailed`:** The failure trace, timeout, or explicit rejection from the infrastructure API.

---

## 7. Specification: Event Structure

Each event in the chain MUST conform to the following schema:

```json
{
  "event_id": "uuid",
  "event_type": "string",
  "intent_id": "uuid",
  "timestamp": "ISO-8601",
  "payload": {
    "description": "Contextual data specific to the event_type"
  },
  "hash": "string"
}
```

*   **event_id**: A universally unique identifier for the specific event.
*   **intent_id**: The correlation ID linking this event to its originating IntentSubmitted event.
*   **payload**: The explicit data (e.g., the Policy Engine output, the Context Snapshot).

---

## 8. Chain Integrity

To ensure tamper-evidence and strict ordering, Events MUST form a hash-linked structure. The cryptographic hash of event $E_n$ is defined as:

$$h_n = \mathcal{H}(E_n.\text{payload} \parallel h_{n-1})$$

Where:
*   $\mathcal{H}$ is a cryptographic hash function (e.g., SHA-256).
*   $\parallel$ denotes concatenation.
*   $h_{n-1}$ is the hash of the immediately preceding event.
*   $h_0$ is a pre-defined genesis value.

This mathematical linkage guarantees tamper detection, verifiable causal ordering, and strict chain integrity.

---

## 9. Replay Semantics

The Replay mechanism allows agents or human operators to reconstruct history.

**Replay MUST:**
1. Process events in strict causal order.
2. Reconstruct system state deterministically using the defined fold function.

**Replay MUST NOT:**
1. Depend on external mutable state or live APIs during reconstruction.
2. Introduce nondeterministic behavior (e.g., relying on random number generators or current system time).

**Replay MAY:**
1. Skip non-critical events (if implementing a fast-forward projection).
2. Operate in "Simulation Mode," re-evaluating historical ContextEvaluated snapshots against updated Policy Engine rules to backtest governance changes.

---

## 10. Context Snapshot Requirements

To guarantee determinism during Replay, Context Snapshots (ContextEvaluated events) MUST:
1. Capture all inputs required for the policy engine to render a decision.
2. Be strictly immutable once written to the chain.
3. Include resource state, active dependency graphs, and relevant external signals (e.g., metrics, logs).

*Note: Context MAY be stored via content-addressed external storage (e.g., IPFS) to prevent payload bloat, provided the reference hash is stored inline.*

---

## 11. Multi-Agent Semantics

The EEC serves as the distributed consensus layer for autonomous actors.

Agents MUST treat the Evidence Chain as the absolute source of truth.
Agents SHOULD NOT rely on local, ephemeral state memory without continuous reconciliation against the EEC.

The chain naturally enables shared understanding, proactive conflict detection, and intent negotiation between disparate agents.

---

## 12. Storage Model

The underlying storage layer for the Evidence Chain MUST:
1. Be append-only.
2. Preserve causal ordering.

Implementations MAY utilize log-based brokers (e.g., Apache Kafka), highly available NoSQL databases (e.g., DynamoDB), or formal immutable ledgers. Partitioning strategies SHOULD guarantee causal ordering per intent_id.

---

## 13. Security Considerations

### 13.1 Historical Tampering
The cryptographic hash chaining defined in Section 8 prevents undetected modification or deletion of historical decisions.

### 13.2 Data Leakage
Context Snapshots may contain sensitive infrastructure data. Implementations SHOULD redact or encrypt sensitive payloads before committing them to the chain, utilizing bounded decryption keys.

### 13.3 Chain Forking
In distributed implementations, network partitions may cause temporary chain forks. Implementations MUST define explicit fork resolution strategies (e.g., Longest Chain, Consensus Voting) to ensure eventual consistency.

---

## 14. Rationale and Alternatives

Traditional CI/CD and infrastructure-as-code systems treat execution as a transient, fire-and-forget action. The Event Evidence Chain redefines execution as a persistent, queryable sequence of decisions.

**Alternatives Considered:**
* **Standard Logging (e.g., ELK Stack):** Rejected due to a lack of schema rigidity, cryptographic linkage, and deterministic replay guarantees.
* **Distributed Tracing (e.g., OpenTelemetry):** Valuable for performance latency, but insufficient for full state-reconstruction and policy reasoning capture.
* **Periodic State Snapshots:** Rejected because they capture the "what" but completely lose the causal "why" (the intent and decision).

---

## 15. Future Work

1. **Cryptographic Signing:** Requiring agents to sign events with zero-knowledge proofs.
2. **Distributed Consensus:** Protocols for cross-cluster chain validation.
3. **Replay Query Language (RQL):** A standardized syntax for querying complex state histories.
4. **Pruning Strategies:** Safe archiving of historical events while maintaining mathematical chain proofs.
